# 《馋嘴猫》小程序开发计划（微信云开发 + 云托管）

## 1. 计划说明

本计划按 **1 名全栈开发 + 1 名兼职产品/运营** 配置估算，目标是在 **6 周完成 MVP，上线灰度版本**。  
如果只有 1 名开发，仍可执行，但运营素材准备和种子内容录入需要同步推进。

## 2. MVP 交付目标

上线时必须满足以下验收条件：

- 用户可浏览首页、榜单、分类、详情页
- 用户可搜索零食并查看推荐理由
- 用户可提交零食/评价/晒图
- 运营可审核内容并决定上架
- 系统可发放和扣减馋豆
- 用户可点击领券购买
- 平台可看到基础数据看板

## 3. 阶段拆解

### 第 0 周：准备阶段（2-3 天）

**目标：** 把开发环境、云资源和项目结构一次性搭好。

**任务清单：**

- 注册并确认小程序主体
- 在微信开发者工具中开通云开发环境
- 创建 `dev/test/prod` 环境
- 初始化代码仓库
- 搭建目录结构：`miniprogram`、`cloudfunctions`、`cloudhosting`、`admin-web`
- 约定分支规范和提交规范
- 配置基础环境变量
- 创建数据库集合和基础索引

**交付物：**

- 可运行的小程序壳工程
- 已创建的云开发环境
- 已部署的云托管空服务
- 数据库集合初始化脚本或说明文档

**验收标准：**

- 本地可打开小程序工程
- 可调用测试云函数
- 云托管健康检查接口可访问

### 第 1 周：基础框架与数据层

**目标：** 完成核心数据表、公共服务层和页面骨架。

**任务清单：**

- 封装小程序请求层、登录态、错误提示
- 完成 `users`、`snacks`、`snack_reviews`、`snack_submissions`、`bean_logs` 等核心集合定义
- 编写 `user-login-init`、`home-feed`、`snack-detail` 基础云函数
- 搭建首页、发现、发布、我的 4 个 Tab 页面骨架
- 设计通用卡片组件、空状态组件、图片上传组件
- 完成基础埋点方案

**交付物：**

- 页面骨架
- 云函数基础能力
- 数据结构文档

**验收标准：**

- 用户首次进入可完成登录初始化
- 首页可返回 mock 数据
- 页面骨架可以完整跳转

### 第 2 周：发现链路开发

**目标：** 跑通用户从浏览到详情的主链路。

**任务清单：**

- 实现首页推荐、榜单、Banner、情绪分类
- 实现搜索结果页、分类页、筛选排序
- 实现零食详情页
- 接通收藏功能
- 增加浏览日志、点击日志
- 准备第一批种子数据，至少 50-100 条

**交付物：**

- 首页
- 搜索和分类页
- 零食详情页
- 收藏能力

**验收标准：**

- 首页到详情页可顺畅跳转
- 搜索可按关键词返回结果
- 收藏状态可正确保存

### 第 3 周：投稿与审核链路

**目标：** 跑通用户贡献内容到运营审核的流程。

**任务清单：**

- 实现发布页
- 支持条码扫码和手动录入
- 支持图片上传到云存储
- 实现 `submission-create`、`review-create`
- 增加文本和图片内容安全预审
- 开发投稿记录页
- 开发后台审核列表和审核详情 API

**交付物：**

- 投稿链路
- 审核链路
- 投稿记录页

**验收标准：**

- 用户可提交新零食或评价
- 审核通过后可写入正式数据
- 驳回内容可展示驳回原因

### 第 4 周：馋豆体系与后台管理

**目标：** 跑通激励和运营最小闭环。

**任务清单：**

- 实现任务中心和馋豆余额展示
- 实现 `bean_logs` 流水
- 实现奖励发放逻辑
- 实现兑换页和兑换记录
- 开发后台奖品管理、Banner 管理、分类管理
- 增加管理员登录态和权限校验

**交付物：**

- 馋豆任务中心
- 兑换页
- 后台配置功能

**验收标准：**

- 审核通过后能自动发放馋豆
- 兑换成功后能正确扣减馋豆
- 运营可在后台修改首页配置

### 第 5 周：云托管服务与 AI/CPS 对接

**目标：** 将复杂能力收敛到云托管，提高后续扩展性。

**任务清单：**

- 初始化云托管服务框架
- 开发 AI 摘要、标签提取接口
- 开发 CPS 链接生成或刷新接口
- 开发数据看板接口
- 开发定时任务：每日榜单生成、异常任务重试
- 完成海报生成接口预留

**交付物：**

- 云托管后台 API
- AI 生成能力
- CPS 对接层
- 日榜同步能力

**验收标准：**

- 审核通过后的零食可异步生成 AI 安利文
- 详情页可读取有效购买链接
- 每日榜单可自动更新

### 第 6 周：联调、测试、灰度上线

**目标：** 完成体验版、灰度版和正式版的上线准备。

**任务清单：**

- 全链路联调
- 修复核心 Bug
- 压测首页、详情、上传、审核接口
- 完成隐私政策、用户协议、客服与反馈页
- 完成基础运营素材录入
- 准备体验版、小范围灰度
- 上线监控与报警

**交付物：**

- 可提审版本
- 上线检查清单
- 初始运营数据

**验收标准：**

- 主流程无阻断 Bug
- 日志可追踪
- 体验版可供真实用户试用

## 4. 工作分解结构（WBS）

| 模块 | 子任务 | 负责人建议 |
| --- | --- | --- |
| 前端小程序 | 页面开发、组件封装、交互实现 | 全栈开发 |
| 云函数 | 鉴权、业务写操作、定时任务 | 全栈开发 |
| 云托管 | AI、CPS、后台 API、聚合服务 | 全栈开发 |
| 运营后台 | 审核、配置、数据看板 | 全栈开发 |
| 数据准备 | 种子零食录入、图片素材整理 | 产品/运营 |
| 规则设计 | 馋豆规则、审核规则、专题规则 | 产品/运营 |

## 5. 具体开发顺序建议

按以下顺序开发，风险最低：

1. 先做环境、数据库、登录和页面骨架。
2. 再做首页、搜索、详情，跑通浏览链路。
3. 再做投稿、审核、发豆，跑通内容生产链路。
4. 最后接入 AI、CPS、数据看板，完善运营和变现链路。

不要反过来先做 AI 或平替推荐，否则很容易陷入“能力很多、主链路没跑通”的状态。

## 7. 数据准备计划

MVP 上线前建议至少准备：

- 100 条零食基础数据
- 200 张真实零食图
- 50 条精选评价
- 8 个首页 Banner/专题位素材
- 4 个情绪分类专题

没有种子数据时，首页会空，推荐体系也不会成立，这是上线最大风险之一。

## 9. 风险与应对

| 风险 | 表现 | 应对方案 |
| --- | --- | --- |
| 冷启动内容不足 | 首页内容稀少、转化差 | 先准备种子数据，先做运营精选 |
| 投稿内容质量差 | 灌水、网图、低质评价多 | 增加审核规则和内容安全校验 |
| AI 成本和时延高 | 页面卡顿、成本失控 | 改为异步预生成，不在用户访问时实时生成 |
| CPS 链接不稳定 | 跳转失败、佣金丢失 | 建立链接刷新机制和点击日志 |
| 审核压力过大 | 投稿积压 | 先做规则拦截，再人工审核 |
| 数据结构频繁变化 | 开发返工 | 第 1 周先冻结 MVP 表结构和字段命名 |

## 10. 上线后两周运营动作

### 第 1 周

- 每日补充 10-20 条精选零食
- 人工精选优质评价和图片
- 观察首页点击率和领券点击率
- 修复首批反馈问题

### 第 2 周

- 上线第一个专题活动
- 启动“首录奖励”运营活动
- 统计高点击低转化零食并优化详情页
- 评估是否进入 V1.1 的平替推荐开发

## 11. 最终建议

这个项目的正确顺序不是先把“AI 推荐”和“爆款玩法”做满，而是先做出一个稳定可运营的内容平台底座。

最小闭环只有 4 个环节：

1. 有内容可看。
2. 有内容可发。
3. 有审核可控。
4. 有领券可转化。

只要这 4 个环节跑通，后续无论是平替推荐、自动种草姬还是更复杂的推荐算法，都可以在现有架构上继续迭代。
