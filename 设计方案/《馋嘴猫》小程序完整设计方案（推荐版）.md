# 《馋嘴猫》微信小程序产品与技术设计方案（落地版）

## 1. 项目概述

### 1.1 产品定位

**产品名：** 馋嘴猫  
**Slogan：** 全网宝藏零食，馋嘴猫带你搜。  
**产品形态：** 微信小程序  
**后台方案：** 微信云开发 + 微信云托管

馋嘴猫是一款聚焦「零食发现 + 用户真实安利 + CPS 转化」的微信小程序。产品通过推荐榜单、真实开箱图、场景化分类和 AI 辅助总结，帮助用户更快找到值得买的零食，并通过优惠券、分销链接和活动任务实现转化。

### 1.2 目标用户

- 喜欢尝鲜、经常囤零食的大学生和年轻白领
- 经常在小红书、抖音、社群中寻找零食推荐的用户
- 愿意分享真实吃后感、晒图和推荐理由的内容型用户

### 1.3 核心价值

- 帮用户省时间：用榜单、标签和推荐理由缩短决策路径
- 帮用户避坑：优先展示真实评价、真实晒图和复购信号
- 帮平台沉淀内容：用馋豆激励用户贡献新零食和真实口碑
- 帮项目变现：用 CPS 领券购买链路完成佣金转化

### 1.4 MVP 目标

MVP 只解决三件事：

1. 用户能快速发现值得买的零食。
2. 用户能贡献真实内容并获得馋豆奖励。
3. 平台能完成审核、上架、领券跳转和基础数据沉淀。

### 1.5 当前版本边界

**MVP 必做：**

- 首页推荐、榜单、分类、搜索
- 零食详情页
- 用户投稿零食、评价、图片
- 馋豆任务和兑换
- 运营审核后台
- CPS 购买跳转
- 基础 AI 总结能力

**V1.1 再做：**

- 平替推荐
- 自动种草海报
- 专题活动页
- 更细的推荐分层

**V2 再做：**

- 个性化推荐
- 社交关注和互动
- 聊天式零食推荐
- 商家侧入驻能力

**当前明确不做：**

- 自建电商交易闭环
- 小程序内支付购买零食
- 复杂 IM 社交系统
- 多端同步发布

## 2. 用户角色与核心链路

### 2.1 用户角色

- **游客：** 可浏览首页、榜单、详情页，但不能投稿和领取任务奖励
- **登录用户：** 可收藏、投稿、评论、上传图片、领取馋豆奖励
- **运营管理员：** 可审核内容、配置首页、管理奖品、处理异常内容

### 2.2 核心用户路径

#### 链路 A：发现并购买

首页/搜索 -> 榜单或分类 -> 零食详情 -> 领取优惠 -> 跳转购买 -> 记录点击

#### 链路 B：发现新零食并投稿

发布页 -> 扫条码或手动录入 -> 上传图片/文案 -> 提交审核 -> 审核通过 -> 奖励馋豆

#### 链路 C：为已有零食补充真实口碑

详情页 -> 写评价/晒图 -> 提交审核 -> 审核通过 -> 奖励馋豆 -> 内容进入详情页展示

## 3. 信息架构与页面规划

### 3.1 Tab 设计

建议采用 4 个一级 Tab：

- **首页：** 推荐、榜单、搜索、专题入口
- **发现：** 分类、情绪场景、平替入口、专题合集
- **发布：** 扫码录入、发评价、上传晒图
- **我的：** 任务、馋豆、收藏、投稿记录、设置

### 3.2 页面清单

| 页面 | 目标 | 核心模块 | 数据来源 |
| --- | --- | --- | --- |
| 首页 | 快速发现值得买的零食 | 搜索框、Banner、猫选之物、无限回购榜、情绪分类 | `home_configs`、`daily_rankings`、`snacks` |
| 搜索结果页 | 按关键词/标签查找零食 | 搜索词、筛选、排序、结果卡片 | `snacks`、`categories` |
| 分类页 | 按场景和品类浏览 | 情绪分类、品类分类、专题合集 | `categories`、`topic_feeds` |
| 零食详情页 | 建立购买意愿并承接转化 | 封面、AI 安利文、必买理由、用户晒图、评价、领券按钮 | `snacks`、`snack_reviews`、`media_assets` |
| 发布页 | 引导用户贡献内容 | 扫条码、手动录入、上传图、写评价、提交 | `snack_submissions` |
| 任务中心 | 提升投稿率和留存 | 任务列表、馋豆规则、完成进度 | `tasks`、`bean_logs` |
| 馋豆兑换页 | 提高激励闭环 | 奖品列表、兑换记录 | `rewards`、`redemption_orders` |
| 我的页 | 账号中心和资产管理 | 个人资料、馋豆余额、收藏、投稿记录 | `users`、`favorites`、`snack_submissions` |
| 投稿记录页 | 反馈审核结果 | 审核状态、驳回原因、奖励记录 | `snack_submissions` |
| 运营后台 | 管理内容和配置首页 | 审核台、榜单配置、Banner 配置、奖品管理、数据看板 | 多集合聚合 |

### 3.3 页面交互重点

- 首页推荐卡片必须在 1 屏内完成「看图 - 看卖点 - 点详情」的路径
- 详情页 CTA 只保留一个主按钮：`领券购买`
- 发布页优先支持 `扫码条形码`，再提供 `手动搜索/补录`
- 上传图片后先走内容安全审核，再进入人工审核队列
- 馋豆奖励不允许前端直接写库，必须由云函数统一发放

## 4. 功能设计

### 4.1 首页：寻宝大厅

**视觉方向：**

- 使用小猫 IP 作为品牌识别点
- 搜索框文案建议为：`搜一下，今天吃点什么好货？`
- 首页首屏只展示 3 类信息：推荐、榜单、场景入口

**模块设计：**

- **猫选之物：** 运营精选 + 推荐指数排序
- **无限回购榜：** 以复购意愿、收藏率、评价质量为核心指标
- **情绪分类：** 加班续命、深夜追剧、约会加分、解压神器
- **专题推荐：** 新手囤货、办公室零食、宿舍回购王

### 4.2 搜索与分类

**搜索能力：**

- 关键词搜索：零食名、品牌名、口味词、热词
- 标签搜索：低卡、微辣、奶香、办公室、追剧
- 条码录入：优先通过 `wx.scanCode` 扫码识别条形码
- OCR 补录：针对包装图、截图等场景辅助识别

**排序方式：**

- 推荐优先
- 热门优先
- 最新收录
- 低价优先

### 4.3 零食详情页：安利笔记区

**核心展示顺序：**

1. 零食封面和标题
2. AI 安利文
3. 3 个必买理由
4. 馋豆专享价和领券按钮
5. 真实晒图
6. 用户评价
7. 同类推荐

**字段说明：**

- **AI 安利文：** 由服务端异步生成，避免每次用户访问实时调用模型
- **必买理由：** 固定 3 条，格式统一，便于卡片化展示
- **真实晒图：** 仅展示已审核通过的用户图
- **领券购买：** 记录点击日志，用于评估转化

### 4.4 发布与内容贡献

**投稿类型：**

- 新零食首录
- 对已收录零食写评价
- 给已有零食补充真实图片

**提交字段：**

- 零食名称
- 品牌
- 条码
- 购买渠道
- 价格区间
- 口味标签
- 内容描述
- 晒图

**审核流程：**

1. 用户提交内容
2. 云函数做基础校验和内容安全预审
3. 进入运营审核队列
4. 审核通过后写入正式集合
5. 触发馋豆奖励和统计更新

### 4.5 馋豆机制

**任务建议：**

| 任务 | 条件 | 奖励 |
| --- | --- | --- |
| 发现宝藏 | 首次录入平台未收录零食并审核通过 | 10 馋豆 |
| 硬核安利 | 提交 30 字以上有效评价并审核通过 | 5 馋豆 |
| 种草秀 | 上传清晰真实晒图并审核通过 | 2 馋豆 |
| 每日签到 | 每日首次打开小程序 | 1 馋豆 |

**兑换建议：**

- 试吃券
- 优惠券模板
- 活动抽奖机会
- 限量周边

### 4.6 运营后台

运营后台不建议直接依赖数据库控制台进行长期运营，至少应具备以下模块：

- 内容审核台
- 零食管理
- Banner 和专题配置
- 榜单配置
- 奖品管理
- 用户与馋豆流水查询
- 数据看板

建议做一个简单 H5 管理后台，前端可部署到云开发静态托管，API 由云托管服务提供。

## 5. 推荐策略与 AI 能力

### 5.1 推荐指数设计

`recommend_index` 建议为 0-100 分，来源于以下维度加权：

- 评价质量分：30%
- 收藏率：20%
- 点击购买率：20%
- 复购意愿词频：20%
- 运营人工加权：10%

### 5.2 AI 使用原则

- AI 只做辅助总结，不作为最终审核结论
- AI 生成内容必须可回溯，保存 prompt 版本和生成时间
- AI 文案采用异步生成，不阻塞主链路
- AI 输出只进入草稿字段，最终发布仍需规则校验或人工审核

### 5.3 AI 适用场景

- 生成安利文
- 提炼 3 个必买理由
- 提取高频关键词
- 计算推荐指数辅助特征
- 自动归类情绪标签
- 生成种草海报文案

## 6. 技术架构

### 6.1 总体架构

```text
微信小程序前端
  ├─ 云开发 SDK
  ├─ 云函数调用
  └─ 云存储上传

微信云开发
  ├─ 云数据库：业务主数据
  ├─ 云存储：图片、海报、素材
  ├─ 云函数：鉴权、写操作、定时任务、轻量聚合
  └─ 静态托管：运营后台前端（可选）

微信云托管
  ├─ 管理后台 API
  ├─ AI 编排服务
  ├─ CPS/三方接口适配层
  ├─ 海报生成服务
  └─ 长耗时任务服务
```

### 6.2 云开发与云托管职责划分

#### 适合放在云开发的内容

- 用户登录态初始化
- 常规 CRUD
- 内容审核后的正式入库
- 馋豆发放和流水记录
- 定时同步榜单
- 图片上传和存储管理

#### 适合放在云托管的内容

- AI 生成和批量异步任务
- 对接 CPS/联盟平台
- 海报生成
- 管理后台聚合接口
- 长耗时导入和批量处理
- 后续可能接入的 Web 管理端 API

### 6.3 推荐技术选型

#### 小程序端

- 原生微信小程序
- TypeScript
- 小程序 npm 能力
- 自定义组件 + 轻量工具库

#### 云函数

- 使用微信云开发当前稳定运行时
- 按业务拆分函数目录
- 统一封装鉴权、日志、异常处理

#### 云托管服务

- Node.js LTS
- Fastify 或 Express
- 以 REST API 形式提供后台服务
- 使用环境变量存储三方密钥

#### 运营后台

- Vue 3 + Vite
- 部署到云开发静态托管
- 通过云托管 API 访问后台能力

### 6.4 环境规划

至少拆成 3 套环境：

- `dev`：开发环境
- `test`：联调与体验版环境
- `prod`：正式环境

每个环境独立配置：

- 云数据库
- 云存储目录
- 云函数环境变量
- 云托管服务配置
- 小程序环境 ID

## 9. 权限、安全与风控

### 9.1 权限原则

- 前端不直接修改馋豆余额
- 前端不直接写正式零食表状态字段
- 所有奖励、审核、上架操作都必须经过云函数或云托管服务
- 管理员角色基于 OpenID 白名单或后台角色表控制

### 9.2 内容安全

- 文本内容接入内容安全检测
- 图片内容接入图片安全检测
- 审核失败内容不进入公开集合
- 重复投稿需提示用户已有零食信息，避免脏数据

### 9.3 数据风控

- 同一用户每日投稿次数限流
- 同一设备/用户短时间内高频提交需触发风控
- 领券点击做去重和日志记录
- AI 结果只缓存入库，不在用户请求时重复生成

## 10. 运维与数据指标

### 10.1 关键业务指标

- DAU / WAU
- 首页推荐点击率
- 详情页到领券点击率
- 投稿通过率
- 用户投稿转化率
- 馋豆消耗率
- 每日新增零食数
- CPS 点击率

## 13. 结论

这个项目最适合采用「微信小程序前台 + 微信云开发做基础后端 + 微信云托管承载复杂服务」的方案。

原因很明确：

- 小程序与云开发天然集成，登录、存储、数据库、云函数成本最低
- 云托管适合承载 AI、运营后台 API、三方 CPS 对接和长耗时任务
- 先做内容发现与转化闭环，再做平替推荐和自动种草，路线最稳

从落地角度看，MVP 不要一开始追求“全网数据抓取”和“强社交”，先用精选内容、用户投稿和榜单机制跑通闭环，后续再放大推荐算法和分发能力。
